## 1. Executive Summary

This audit checked every one of WorkJax's 38 hard-coded employer records to identify, from official sources only, which applicant-tracking system (ATS) or career-site platform each one uses, and whether that platform could plausibly power a second `docs/integrations/employer-feed-registry.md`-style live feed the way Dun & Bradstreet's Lever board does today.

**One clear, confirmed candidate emerged: Fanatics (`id: 19`).** Fanatics' careers site is confirmed, via direct Greenhouse-hosted job-application URLs under its own name (`job-boards.greenhouse.io/fanaticsinc`), to run on **Greenhouse** â€” the same class of platform already proven out for Lever with Dun & Bradstreet, with a comparably well-documented public, unauthenticated Job Board API (`docs/integrations/api-evaluation.md`, Section 5.8). Fanatics is also a genuine, large Jacksonville employer (a real Southpoint-area office, already curated in `data.js` with real internship programs), not merely a company that happens to use a supported ATS.

**No other employer in the current dataset was confirmed on Lever, Greenhouse, or SmartRecruiters.** A handful of other platforms were identified with reasonable-to-strong confidence â€” Taleo (JEA and JEA Skilled Craft Apprenticeship), Workday (VyStar Credit Union, Fidelity Investments), iCIMS (UF Health Jacksonville, and likely for Intercontinental Exchange), TeamWork Online (Jacksonville Jaguars, both records), and a cluster of Oracle Fusion Cloud Recruiting instances (Mayo Clinic Florida, Nemours Children's Health, CSX Transportation) â€” but **none of these platforms has a confirmed, publicly documented, unauthenticated read endpoint** comparable to Lever's or Greenhouse's Postings/Job Board APIs. Several employers (WJCT Public Media, the State Attorney's Office, the Electrical Training Alliance of Jacksonville, the NEFBA Apprenticeship program, and St. Johns Riverkeeper) are confirmed to be **manual-application-only** today (email, in-person, or a paper form), with no ATS at all. A large remaining group could not be confirmed on any platform from official sources within this session's tooling limits and are marked `NOT VERIFIED`.

**A significant tooling limitation affected this entire audit and is disclosed in full in Section 2:** this session's `WebFetch` tool returned HTTP 403 for every URL attempted, including non-employer control URLs like Wikipedia â€” a total block, not a per-site policy decision. All findings below rest on `WebSearch` results, restricted to cases where the search result's own URL is itself the official ATS-hosted page (e.g., a `job-boards.greenhouse.io/<company>` or `<company>.taleo.net` URL appearing as a real, named result) â€” never on a third-party description, blog, or aggregator's characterization of what platform a company "probably" uses. Every finding below carries an explicit confidence label, and every employer that could not be pinned to a specific platform this way is labeled `NOT VERIFIED`, per the task's evidence rules.

## 2. Method and Evidence Standards

### Sources actually used
- Each employer's own official careers/internship page (generally the same URL already curated as `internshipUrl` in `data.js`).
- Where a search surfaced it, the **ATS-hosted job-board page itself** â€” e.g. `jobs.lever.co/*`, `job-boards.greenhouse.io/*`, `*.myworkdayjobs.com`, `*.icims.com`, `*.taleo.net`, `*.oraclecloud.com/hcmUI/*`, `teamworkonline.com/*` â€” treated as primary evidence only when the URL itself appears under the employer's own name/board-token/site-slug.
- Official ATS developer documentation already vetted in `docs/integrations/api-evaluation.md` (Greenhouse: `developers.greenhouse.io`; Lever: `hire.lever.co/developer/postings`).
- Official government pages for public-agency employers (City of Jacksonville, JEA, JTA, JAA, the State Attorney's Office, JAXPORT).
- The existing, already-validated D&B/Lever integration itself (`api/dnb-lever-jobs.js`) as direct evidence for `id: 41` â€” no new research was needed there.

### A tooling limitation that shaped every finding in this document
Within this session, `WebFetch` (direct page retrieval) returned **HTTP 403 Forbidden for every URL attempted this session, without exception** â€” including employer career pages, ATS-hosted pages, and a Wikipedia control URL used specifically to test whether the block was employer-specific or total. It was total. This matches, and is more restrictive than, the network limitation `docs/integrations/api-evaluation.md` already documented for a subset of domains in an earlier session.

Because of this, **no page in this audit was directly, independently loaded and read**. Every finding instead relies on `WebSearch`, which did function normally. To keep this consistent with the task's rule against "search-result snippets as final evidence," this audit applies a stricter standard than a simple snippet read:

- **Confirmed** is used only when a search result's own URL is a live-looking, employer-named page hosted directly on the ATS's own domain (e.g., `job-boards.greenhouse.io/fanaticsinc`, `jea.taleo.net/careersection/jea+-+external/...`), or when the organization's own text explicitly names its process (e.g., WJCT's "email, fax, or mail" instructions, quoted directly).
- **Likely** is used when the evidence is a strong but singular signal (e.g., one Oracle Cloud recruiting URL tied to the employer's name appearing once) that a direct page load would have made trivial to fully confirm, but which this session's tooling could not complete.
- **Uncertain** is used for weaker, indirect, or conflicting signals (e.g., a job listing that mentions "Workday" only because the posting is *for* a Workday-administration role, not because it demonstrates the employer's own careers site runs on Workday).
- **`NOT VERIFIED`** is used, per the task's explicit instruction, whenever no search result surfaced a platform-identifying URL or explicit statement at all.

**This audit's confidence should be re-verified with direct page access (a different tool, a different session, or a manual check) before any implementation decision is finalized** â€” this document identifies candidates and rules out obvious non-candidates; it does not substitute for a direct technical spike against the actual endpoint, which Section 11 defines as the required next step.

### Provider categories used
Lever, Greenhouse, SmartRecruiters, Workday, iCIMS, Taleo, SuccessFactors, Government/custom, Other (named specifically â€” e.g. Oracle Fusion Cloud Recruiting, TeamWork Online, SilkRoad, Volgistics, Galaxy Digital, iamempowered.com), or `NOT VERIFIED`.

## 11. Proposed Next Proof-of-Concept Scope

Following the same pattern already used for Dun & Bradstreet (`docs/integrations/dnb-lever-poc.md`, `docs/decisions/ADR-003-dnb-lever-job-poc.md`) and recommended generally in `docs/integrations/api-evaluation.md` Section 11:

1. **A small, local, throwaway, read-only script** (not a Vercel Function, not committed to the deployed site, no registry entry, no API key) calls Greenhouse's public Job Board API for both `fanaticsinc` and `fanaticscollectibles` board tokens.
2. **Confirm which board token(s), if either, currently return any Jacksonville-located postings**, using the same text-based location-matching approach already proven for D&B's Lever `categories.location` field â€” do not assume Jacksonville content exists just because Fanatics has a Jacksonville office.
3. **Confirm which qualifying postings, if any, are genuinely student/early-talent-relevant** (internship, co-op, early-career program) versus ordinary full-time roles, using the same conservative, structural-signal-based classification approach documented in `docs/integrations/dnb-lever-poc.md` (title/department/commitment matching, word-boundary safe) â€” not a blanket "anything Fanatics posts" approach.
4. **Manually cross-check the raw Greenhouse response** against 2â€“3 of Fanatics' currently-curated `data.js` programs (Software Engineering Intern, Data Analytics Intern, Merchandising Intern, Supply Chain Intern) to see whether any live posting maps to them.
5. **Document field-level fit** against the same normalized shape already used for D&B (`title`, `postingKind`-equivalent classification, `location`, `workplaceType`-equivalent if present, `salaryRange`-equivalent if present, `applyUrl`) â€” Greenhouse's list endpoint does not expose salary or workplace-type fields as standard, per `docs/integrations/api-evaluation.md` Section 5.8, so this spike should explicitly confirm what is and isn't actually present in Fanatics' real responses, not assume parity with Lever's richer schema.
6. **This spike does not touch `data.js`, `app.js`, `live-opportunity-sources.js`, or any other website file, requests no API key, adds nothing to the registry, and is fully reversible** â€” deleting the script removes all trace of the experiment, exactly as the D&B/Lever and Ticketmaster spikes were scoped.
7. **Success criteria:** at least one real, current, Jacksonville-tagged, genuinely student/early-talent-relevant Fanatics posting is returned with a usable title, location, and official `absolute_url` â€” mirroring the bar D&B's own proof of concept had to clear.
8. **If the spike finds zero qualifying Jacksonville postings** (the same outcome D&B's own proof of concept partially had â€” its one qualifying result was a Talent Network signup, not an open job), that is itself a valid and important finding, not a failure â€” it would mean Fanatics' Greenhouse board is technically integrable but currently has nothing student-relevant and Jacksonville-specific to show, which should be documented rather than hidden.

**Answering the six required points from the task:**

1. **Best second employer for a proof of concept:** Fanatics (`id 19`).
2. **Backup employer:** No confirmed equal-tier backup exists; UF Health Jacksonville (`id 8`) is the most plausible next candidate, conditional on a not-yet-done iCIMS public-access-model check.
3. **Likely provider adapter needed:** A Greenhouse Job Board API adapter â€” structurally similar to the existing Lever adapter in `api/dnb-lever-jobs.js`, but built fresh, per this task's restriction against modifying any existing API function.
4. **Can the existing normalized D&B response shape be reused?** Partially. The top-level envelope (`source`, `employer`, `generatedAt`, `count`, `jobs`) and several per-job fields (`id`, `title`, `location`, `applicationUrl`) map cleanly. Greenhouse's list endpoint does **not** natively provide `salaryRange`, `workplaceType`, or `commitment` the way Lever does â€” those fields would need to come back as `null`/absent for a Greenhouse-sourced job, exactly like the existing frontend (`liveOpportunityCardHTML` in `app.js`) already handles optional fields today. `postingKind` (open opportunity vs. talent network) has no Greenhouse equivalent field and would need its own conservative, title/description-based classification logic, mirroring â€” not reusing verbatim â€” the approach `api/dnb-lever-jobs.js` uses for Lever's "Early Talent Network" postings.
5. **Additional fields or classifications needed:** A Fanatics-specific "is this actually a Jacksonville, student-relevant, currently-open posting" filter (analogous to `isJacksonvilleFloridaPosting`/`isStudentRelevantPosting`/`classifyOpportunityType` in `api/dnb-lever-jobs.js`, but written fresh for Greenhouse's field names/shape) â€” this is explicitly new work, not a reuse of D&B's filtering functions, since Greenhouse's `location`/`department`/`metadata` fields differ from Lever's `categories`.
6. **Can the integration be implemented without a secret?** Yes â€” Greenhouse's Job Board API GET endpoints are public and unauthenticated, per official documentation already reviewed in `docs/integrations/api-evaluation.md`. No API key would be required, consistent with this task's restriction against creating one.
7. **Exact validation required before implementation:** (a) live-read both Fanatics Greenhouse board tokens and manually confirm real, current Jacksonville-tagged content exists; (b) confirm which board token is the correct match for WorkJax's existing `id 19` record (or whether both should be considered and how to avoid a duplicate card); (c) manually cross-check a sample of returned postings against the classification logic before trusting it, the same way `docs/integrations/dnb-lever-poc.md`'s manual testing checklist required for D&B; (d) re-confirm, with working direct page access (not just search snippets), the exact Greenhouse list-endpoint field set actually returned today, since API surfaces can drift from documentation over time.

## 12. Open Questions and Limitations

- **This session's `WebFetch` tool was completely non-functional** (100% HTTP 403, including a Wikipedia control test) â€” every finding in this document rests on `WebSearch` results whose own result-URL was the official ATS-hosted page, per the stricter standard defined in Section 2. **A follow-up pass with working direct page access should re-verify every `likely`-confidence finding in Section 3a before any of them is treated as settled** â€” `likely` is this audit's ceiling of confidence under the tooling available, not the same as `confirmed`.
- **The Fanatics recommendation itself is unverified at the data level** â€” this audit confirmed the *platform* (Greenhouse) and the *access model* (public GET), not the *content* (whether real, current, Jacksonville-tagged, student-relevant postings actually exist on the live board today). Section 11's spike is required before any registry entry would be justified.
- **Fourteen employers remain `NOT VERIFIED`** (`id`s 10, 13, 14, 15, 17, 18, 21, 24, 29, 30, 34, 35, 37, 39) with no platform identified at all from official sources within this session's constraints â€” these are not "confirmed to have no ATS," they are simply unconfirmed either way, and should not be assumed to be either automatable or non-automatable.
- **Two pre-existing employer-duplicate pairs materially affect candidacy:** JEA/JEA Skilled Craft Apprenticeship (`id 3`/`id 31`) and the two Jacksonville Jaguars records (`id 23`/`id 40`), both already flagged in `docs/data/date-normalization-audit.md` and `docs/data/migration-map.md` as a data-quality issue independent of this audit. Neither pair could become a live-feed candidate without first resolving which `data.js` record is canonical, to comply with this task's (and the registry's) explicit prohibition on a duplicate employer card.
- **A meaningful share of the audited organizations are not job/internship postings systems at all** â€” several are volunteer-shift schedulers (Galaxy Digital, Volgistics) or multi-employer placement aggregators (iamempowered.com, MYAWP) â€” these were included per the task's instruction to audit every existing `data.js` employer, but they are a structural mismatch for the registry's single-employer live-feed model regardless of any future API access, and that mismatch is itself a finding worth preserving here rather than re-discovering later.
- **No employer outside WorkJax's current 38-record `data.js` dataset was evaluated or suggested**, and this document does not recommend adding one, per the task's explicit restriction.
- **This document does not implement anything.** No registry entry was added to `live-opportunity-sources.js`, no API function was created or modified, and no code in `index.html`, `styles.css`, `app.js`, or `data.js` was touched.
